## 重構建議（依優先級排序）
### 🔥 高（改了明顯受益）
- 先清死碼，而不是整檔重寫。未使用 class 有 8 個，其中 7 個都集中在 2026-04 award 疊加層；整段 award 區塊從 [assets/site.css](/E:/ai_website/goodjob-site/assets/site.css:1256) 開始，共 152 行、約 3891 bytes raw，約佔整檔 15.2%，但真正活著的只有 `.award-badge` 與 `.hero-award`。
- 把金色獎項樣式抽成 CSS variables 或共用 class。完全相同的金色主漸層 `linear-gradient(135deg, #ffe28a 0%, #f5c645 45%, #d89a18 100%)` 在 [1262](/E:/ai_website/goodjob-site/assets/site.css:1262)、[1317](/E:/ai_website/goodjob-site/assets/site.css:1317)、[1347](/E:/ai_website/goodjob-site/assets/site.css:1347)、[1387](/E:/ai_website/goodjob-site/assets/site.css:1387) 重複 4 次；同一組金色 border / glow 也散在多個 selector。現在整檔只有 8 個變數、僅 12 個 declaration 使用 `var(...)`，但有 149 處直接寫色碼或 `rgba(...)`。
- 補回結構分段，再做小步整理。`Responsive` 區塊在 [1139](/E:/ai_website/goodjob-site/assets/site.css:1139)，但最新 award 區塊又追加在其後；這代表檔案已開始失去「base / layout / components / utilities / responsive」的清楚順序。另有 `.poster-card, .landscape-card` 在 [430](/E:/ai_website/goodjob-site/assets/site.css:430) 與 [1257](/E:/ai_website/goodjob-site/assets/site.css:1257) 分兩次宣告，維護時容易漏看。

### 🟡 中（可延後）
- 合併已經被證明重複的宣告群。例子：`.topbar-tools` 與 `.detail-gallery-head-right` 都是 `display:flex + align-items:center + gap:16px`（[112](/E:/ai_website/goodjob-site/assets/site.css:112)、[754](/E:/ai_website/goodjob-site/assets/site.css:754)）；`.shelf-arrow:hover` 與 `.detail-gallery-arrow:hover` 都只改同一個 hover 背景（[301](/E:/ai_website/goodjob-site/assets/site.css:301)、[869](/E:/ai_website/goodjob-site/assets/site.css:869)）；兩個 `[hidden]` 規則都寫成 `display: none !important`（[692](/E:/ai_website/goodjob-site/assets/site.css:692)、[948](/E:/ai_website/goodjob-site/assets/site.css:948)）。
- 命名風格不用全面改成 BEM，但要停止再新增 generic class。現況 112 個 class 幾乎全是 kebab-case，沒有 BEM，也沒有 camelCase；一致性其實不差。問題在 generic globals 太多，例如 `.wrap`、`.nav`、`.button`、`.hero`、`.toast`、`.brand`、`.topbar`、`.shelf`、`.lightbox`。這些名稱在更多頁面共享時容易撞名。
- Responsive 邏輯目前不算散。整檔只有 2 個 `@media` 區塊（[1141](/E:/ai_website/goodjob-site/assets/site.css:1141)、[1208](/E:/ai_website/goodjob-site/assets/site.css:1208)），算是集中管理；但 award 區塊追加在 responsive 後面，代表後續若再補 mobile 樣式，極可能開始散落。

### 🟢 低（可選）
- 清理小型冗餘。`.award-badge` 同一個 rule 內重複宣告 `border`，而 `font-family` 先寫字型後又被 `font-family: inherit` 蓋掉；這種問題雖不影響功能，但會降低可讀性。
- 正規化色碼寫法。整檔同時存在 `#fff` 與 `#ffffff`；這類不一致不急，但整理後可讀性會更好。
- 不必為性能把它拆成多個 runtime CSS。`site.css` gzip 後只有約 5.1 KB，拆檔本身不會帶來明顯載入收益。若真的要拆，建議拆「來源結構」再編譯回單一輸出檔，而不是直接改成多檔上線。

## 風險評估
- 不建議整檔重寫。`assets/site.js` 會動態建立多個 class：`.top10-card` / `.top10-rank` / `.poster-card`（[102](/E:/ai_website/goodjob-site/assets/site.js:102) 到 [109](/E:/ai_website/goodjob-site/assets/site.js:109)）、`.award-badge`（[137](/E:/ai_website/goodjob-site/assets/site.js:137)）、`.landscape-card`（[162](/E:/ai_website/goodjob-site/assets/site.js:162)）、`.detail-thumb`（[600](/E:/ai_website/goodjob-site/assets/site.js:600)）、`.shelf-arrows` / `.shelf-arrow`（[821](/E:/ai_website/goodjob-site/assets/site.js:821) 到 [830](/E:/ai_website/goodjob-site/assets/site.js:830)）。一旦改 selector 名稱、順序或 specificity，最容易壞的是 hover preview、popup、detail modal 與 rail arrows。
- 風險是「互動穩定性」，不是「跨站波及」。因為 `site.css` 只影響 `index.html`、`muse-2026.html`、`workflow.html`，不會直接動到 `teabar.html`、`admin/index.html`、`wedding-packages/index.html`。
- 若 silver / bronze 徽章是未來資料模型預留，`award-badge-silver`、`award-badge-bronze` 可以先標記為 dormant，不一定要立刻刪；但目前它們確實沒有任何 HTML/JS 引用。
- 最保守的做法是三段式：先刪死碼，再抽 variables / 共用樣式，最後才整理分段。每一步都只做小量改動並驗 `index` / `workflow` / `muse-2026` 的桌機與手機版。

## 結論與下一步
- 先做一輪低風險清理：刪 8 個未使用 class，順手移除 `.award-badge` 內的重複屬性，這一步幾乎只有收益，風險最低。
- 第二步只做「抽 token，不改命名」：至少把 award 金色主漸層、border、glow 抽成 3 到 5 個變數，先解掉 4 次重複漸層與多處硬編碼色值。
- 暫時不要為了性能拆多檔；若後續還會長功能，才考慮把來源拆成 `base / components / award / responsive`，但上線仍維持單一 `assets/site.css`。

# 村山良作 R2 CDN 維運手冊

> 建立日期：2026-04-17
> 適用 commit：`6319364` 起
> 維運對象：goodjob.weddingwishlove.com

## 一、總覽

村山良作作品圖從本機 `assets/images/` 遷移到 Cloudflare R2，透過 custom domain `goodjob-img.weddingwishlove.com` 對外。目的：
- 解決兩地開發圖片不同步（本機不再需要 clone 2.3GB 圖片）
- 壓縮儲存（2.3GB → 252 MiB，-89%）
- CDN 全球加速、出口流量免費

---

## 二、R2 架構

| 項目 | 值 |
|------|------|
| Bucket | `goodjob-images`（APAC 區）|
| Custom domain | `goodjob-img.weddingwishlove.com`（一層深 subdomain）|
| S3 endpoint | `https://45c5b38d200f8783f05e5957f807b86f.r2.cloudflarestorage.com/goodjob-images` |
| 最小 TLS | 1.2 |
| Public access | 透過 custom domain（公用開發 URL 未啟用）|

### URL 結構

```
https://goodjob-img.weddingwishlove.com/works/{article.id}/{classified}.webp
```

檔名分類規則（見 `migrate_to_r2.py` 的 `classify_filename`）：
1. `{prefix}-hero/detail-N/scene-N.ext` → `{hero/detail-N/scene-N}.webp`
2. `{article_id}_{rest}` → `{rest}.webp`（CMS 上傳風格）
3. `{8hexhash}_{rest}` → `{rest}.webp`（舊 CMS 上傳）
4. 其他 → 檔名 sanitize 保留

### Bucket 結構

```
goodjob-images/
└── works/
    ├── lativ-magic-platform/
    │   ├── hero.webp
    │   ├── detail-1.webp … detail-N.webp
    │   └── scene-1.webp … scene-N.webp
    ├── {article_id}/ …（27 個作品資料夾）
```

## 三、部署架構

### 檔案職責

| 檔案 | 用途 |
|------|------|
| `server.py` | HTTP server + REST API + `/api/upload/{id}` 走 R2 |
| `data/articles.json` | 文章資料（圖片欄位是 R2 完整 URL）|
| `migrate_to_r2.py` | 批次轉檔 + 上傳（一次性，新增作品不需再跑）|
| `rewrite_paths.py` | 依 `path-map.json` 改寫 JSON/HTML 引用 |
| `cleanup_migrated.py` | 刪除已上 R2 的本機原檔（驗證後釋放空間）|
| `upload_asset.py` | 單檔上傳 helper（非 CMS 的靜態圖片用，例新頁面插圖）|
| `path-map.json` | 遷移時產生的本機 → R2 URL 映射（回滾時需要）|

### systemd 設定

Service：`murayama-goodjob.service`
Override：`/etc/systemd/system/murayama-goodjob.service.d/r2.conf`

```ini
[Service]
Environment="GOODJOB_RCLONE_BIN=/home/achilean/bin/rclone"
Environment="GOODJOB_R2_REMOTE=r2:goodjob-images"
Environment="GOODJOB_CDN_DOMAIN=https://goodjob-img.weddingwishlove.com"
Environment="GOODJOB_WEBP_QUALITY=90"
Environment="HOME=/home/achilean"
```

**改 domain 或 bucket 時改這裡**，不要動 `server.py` 預設值。

### rclone 設定

位置：`/home/achilean/.config/rclone/rclone.conf`

```ini
[r2]
type = s3
provider = Cloudflare
access_key_id = <R2 Access Key>
secret_access_key = <R2 Secret>
endpoint = https://45c5b38d200f8783f05e5957f807b86f.r2.cloudflarestorage.com
region = auto
acl = private
```

**憑證輪替：** dashboard → Manage R2 API Tokens 產新的，編輯 `rclone.conf`，重啟 service。

## 四、依賴

| 依賴 | 用途 | 安裝 |
|------|------|------|
| Python stdlib | server.py 主體 | 內建 |
| Pillow | WebP 轉檔 | `pip install --user Pillow`（server）|
| pillow-avif-plugin | AVIF 偽裝 jpg 解碼 | `pip install --user --break-system-packages pillow-avif-plugin` |
| pillow-heif | HEIC（保險，目前沒用到）| 同上 |
| rclone | R2 上傳（subprocess）| `~/bin/rclone`，binary 直裝 |

### 為什麼需要 pillow-avif-plugin

遷移時發現 26 張 jpg/png 其實是 AVIF 格式偽裝（用戶從 iOS 手機上傳未正確轉檔）。Pillow 預設不支援 AVIF，會拋 `cannot identify image file`。plugin 只要 `import pillow_avif` 就自動註冊，不用其他 API 改動。

## 五、新增作品流程（admin CMS）

使用者在 `/admin/` 建立文章並上傳圖片時：

1. 前端 POST `/api/upload/{article_id}` multipart/form-data
2. `server.py` 收到：
   - 讀檔 → Pillow 轉 WebP (q=90, method=6)
   - 透過 `_classify_upload_name()` 命名
   - 寫入 `/tmp/` 暫存
   - 呼叫 `rclone copyto` 上 `r2:goodjob-images/works/{article_id}/{name}.webp`
   - 刪除暫存
3. 回傳 R2 CDN URL，塞進 article.images[] 存回 articles.json

**失敗模式**：
- Pillow 不可用 → 原 bytes 直接上 R2（副檔名不變成 .webp）
- rclone 失敗 → 500 錯誤，未寫入 articles.json，無殘留

## 六、常見問題與對策

### SSL 卡住 pending 很久

**症狀**：dashboard 顯示「使用中」但 curl TLS alert 40
**原因**：二層深 subdomain（例 `img.goodjob.weddingwishlove.com`）不在 CF 免費 Universal SSL 覆蓋範圍
**對策**：用一層深（`goodjob-img.weddingwishlove.com`），SSL 秒通

### admin 上傳回 500

**檢查清單**：
1. `sudo systemctl status murayama-goodjob` 看 stderr 有沒有 `[r2-upload]` 錯誤
2. 確認 systemd override 的 `GOODJOB_RCLONE_BIN` 指向對的 binary
3. `sudo -u achilean /home/achilean/bin/rclone ls r2:goodjob-images | head` 測憑證

### 某張圖讀不到 404

**可能原因**：
- `articles.json` 有 URL 但 R2 沒檔（migration 漏掉或檔名分類錯）
- 對應方法：`rclone ls r2:goodjob-images/works/{slug}/` 對比 articles.json，缺的用 `migrate_to_r2.py --article {slug}` 補

### CF 快取頑固

- 編輯 `site.css` / `site.js` 後記得升版 `?v=YYYYMMDD[字母]`（既有慣例）
- R2 的圖因為 URL 是不變的，快取 OK，不需要版本字串

## 七、回滾到本機路徑

萬一 R2 服務中斷且需要立刻回退：

```bash
cd 村山良作
python3 rewrite_paths.py --restore  # 從 .bak 還原 articles.json 與 HTML
```

但前提：
- `.bak` 檔還在（`rewrite_paths.py` 首次跑時建立的）
- 主機 `assets/images/` 原檔還在（遷移腳本不刪原檔，應該還在）

若要 hard revert，`path-map.json` 是反查表，可以寫個反向腳本把 R2 URL 全改回本機路徑。

## 八-5、新增頁面用圖（非作品）— `upload_asset.py`

若你在寫新的 HTML 頁面或 section，需要一張插圖（非 admin CMS 管理的作品圖），用 `upload_asset.py` 直接上 R2，避免再把圖留本機需要手動 scp。

### 流程

```bash
# 1. 本機圖 scp 上主機
scp my-banner.png achilean@100.102.51.64:/tmp/

# 2. 主機上轉 WebP + 上傳 + 拿 URL
ssh achilean@100.102.51.64 \
  'python3 /srv/weddingwish/goodjob-sit/upload_asset.py /tmp/my-banner.png'
# 輸出：https://goodjob-img.weddingwishlove.com/static/my-banner.webp

# 3. 貼到 HTML/CSS
<img src="https://goodjob-img.weddingwishlove.com/static/my-banner.webp">

# 4. 只 scp HTML 到主機 + 重啟，**圖不用管**
```

### 常用 flags

```bash
--prefix banners     # 換 bucket 下的資料夾（預設 static/）
--name hero.webp     # 覆寫目的檔名
--quality 95         # 提高品質（預設 90）
--no-convert         # 不轉 webp（給已優化的 .webp/.svg/.gif）
```

### 原則

- **作品圖（articles.json 管的）走 admin CMS**，不用這個 helper
- **UI chrome（logo/favicon 這類低頻變動）**保留本機，不用這個
- **新頁面的插圖、banner、promo image** 適合這個

## 八-7、清理本機原檔 — `cleanup_migrated.py`

遷移完畢確認線上穩定後，清掉主機 `assets/images/` 已上 R2 的原檔釋放空間。

### 安全特性

- 只刪 `path-map.json` 裡有對應 R2 URL 的檔案
- 每刪一檔前先 `rclone lsf` 驗 R2 有對應 object（`--no-verify` 可跳過）
- 未在 path-map 的檔案（logo、favicon、teabar 等）**永遠不會被刪到**

### 流程

```bash
# 單篇測試（強烈建議）
python3 cleanup_migrated.py --article renai-civil --dry-run
python3 cleanup_migrated.py --article renai-civil

# 確認線上仍正常後，逐篇或一次全清
python3 cleanup_migrated.py --all --dry-run
python3 cleanup_migrated.py --all

# 信任批次時可 skip 每檔 R2 驗證（快很多）
python3 cleanup_migrated.py --all --no-verify
```

## 八-8、清理孤兒檔 — `cleanup_orphans.py`

`cleanup_migrated.py` 只清「有在 path-map 的」檔。但 `assets/images/` 還有很多**沒在 articles.json 裡引用**的歷史檔（歷代 CMS 上傳、舊版素材、test 圖）。用 `cleanup_orphans.py` 掃描全站源碼，保留真的有被引用的，其餘搬進 `assets/images-trash/` 隔離區。

### 演算法

1. 掃 `SRC_ROOT` 底下所有 `.html/.css/.js/.py/.md/.json/.xml/.txt`
2. Regex 抓 `/assets/images/{path}.{ext}` 建立 refs set
3. 讀 `path-map.json` keys 也納入保護（雙保險）
4. 掃 `assets/images/` 所有實體檔
5. 不在保護 set 的 → orphan → 搬 `assets/images-trash/`

### 流程

```bash
# 預覽
python3 cleanup_orphans.py

# 搬去 trash（可復原）
python3 cleanup_orphans.py --apply

# 需要時復原
python3 cleanup_orphans.py --restore

# 觀察期過了真的刪
python3 cleanup_orphans.py --purge
```

### 執行紀錄

- 2026-04-17: 搬 696 張（~560MB）進 trash，線上 goodjob + teabar 正常。`teabar/brand-event.jpg` 正確保留（teabar.html 有引用）

### 注意事項

- 掃描不涵蓋動態路徑（`fetch('/assets/images/' + name)`）— 若有這種用法要手動將資料夾加白名單
- Regex 匹配必須有**開頭斜線**（`/assets/images/`）。若 HTML/CSS/JS 用 `assets/images/foo.jpg`（無斜線相對路徑）會漏抓 — 目前此專案未見此寫法
